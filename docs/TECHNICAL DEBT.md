# Technical Debt & Deferred Items

This file tracks implementation items that were **identified during development but consciously deferred** — typically because the risk was assessed as manageable at the current phase, or the prerequisite work was not yet in place.

This is distinct from the feature backlog (`docs/implementation/V1 IMPLEMENTATION BACKLOG.md`), which tracks planned features. Items here are retrospective findings from code reviews and assessments.

**Process:**
- When a code review, assessment, or implementation reveals a gap that will not be fixed immediately, add an entry below.
- When an item is resolved, update its `Status` to `Resolved — <phase/date>` and optionally note what was done.
- Entries are never deleted — resolved items stay for audit history.

---

## Entry Format

```
## DEBT-NNN — Short Title
- **Source:** Which review/assessment identified this
- **Risk:** Low / Medium / High — one-line description of what breaks if left unaddressed
- **Affected Files:** List of relevant files
- **Why Deferred:** Explicit reason (assessment guidance, out of scope, prerequisite not ready, etc.)
- **Target:** Suggested phase or condition under which to resolve
- **Status:** Open / Resolved — <phase/date>
```

## DEBT-001 — Concurrency Control for Allocation Engine

- **Source:** Phase A & B Assessment ⚠️3
- **Risk:** High — two simultaneous payment allocations for the same member can both fetch the same open obligations, leading to over-allocation and potentially negative `amount_outstanding` on obligations
- **Affected Files:**
  - `chama/chama_contributions/services/allocation_engine.py` (`allocate_payment`)
- **Why Deferred:** Assessment explicitly said "soon, not now". The risk is real but low-probability in the current single-server, low-concurrency deployment. The allocation engine is logically correct; it is not race-condition-safe.
- **Target:** Resolve before Phase C. Loan disbursements introduce a second class of concurrent financial mutations that will compound this risk.
- **Suggested Fix:** Add row-level locking (`SELECT ... FOR UPDATE`) on the obligation rows fetched by `get_open_obligations()`, or wrap `allocate_payment()` in a Frappe `enqueue`-based serial queue per member.
- **Status:** Open

## DEBT-002 — Fuzzy Duplicate Payment Detection

- **Source:** Phase A & B Assessment ⚠️5
- **Risk:** Medium — duplicate payments can slip through if the member uses a different payment reference each time (e.g., two M-Pesa transactions for the same amount, same day, same member). Current detection only flags exact reference matches within the same Chama.
- **Affected Files:**
  - `chama/chama_contributions/doctype/chama_contribution_payment/chama_contribution_payment.py` (`_check_duplicate_reference`)
- **Why Deferred:** Assessment said "not now". Implementing fuzzy detection before Phase D (disbursements) risks false positives that block legitimate payments. The current reference-match flag is sufficient for v1 operator review workflows.
- **Target:** Phase D or E. By then the payment volume and operator tooling will make the cost of false positives vs. missed duplicates easier to calibrate.
- **Suggested Fix:** After inserting a payment, run a secondary check: same Chama + same member + same `amount_received` ± 0 + payment date within a configurable rolling window (default: 24 hours). Flag but do not block. Expose a "Review Duplicates" report for operators.
- **Status:** Open

## DEBT-003 — Penalty Obligation System (Beyond Skeleton)

- **Source:** Phase A & B Assessment ⚠️6 / Phase B design
- **Risk:** Medium — the `apply_penalties_skeleton()` scheduler job identifies overdue obligations eligible for penalty but creates no penalty obligation records. Until penalties are implemented, members face no financial consequence for late payment beyond the "Overdue" status label.
- **Affected Files:**
  - `chama/chama_contributions/services/obligation_status_jobs.py` (`apply_penalties_skeleton`)
  - `chama/chama_contributions/doctype/chama_contribution_obligation/chama_contribution_obligation.json` (needs `penalty_applied` field wired to logic)
- **Why Deferred:** Phase B scope explicitly deferred penalties. The skeleton and `penalty_applied` flag exist as placeholders. Penalty rules (flat fee vs. percentage, compounding, grace periods) require input from the Chama settings design.
- **Target:** Phase C or D. Penalty categories should reuse `Chama Contribution Category` with a `category_type = Penalty` option. Penalty obligations are auto-generated by the scheduler and allocated against like any other obligation.
- **Suggested Fix:**
  1. Add `penalty_rate`, `penalty_type` (Flat / Percentage), `penalty_compounding` to `Chama Settings`.
  2. In `apply_penalties_skeleton()`, replace the log-only loop with obligation creation using those settings.
  3. Mark `penalty_applied = 1` on the source obligation after creating the penalty obligation.
- **Status:** Open
